# 涨幅榜功能产品需求文档

## 1. 产品概述

### 1.1 功能描述
涨幅榜是Ave.ai平台的核心功能之一，用于展示各个区块链上代币的涨幅排名数据。该功能将完全参照现有热搜榜的实现思路，提供实时的代币涨幅数据展示和筛选功能。

### 1.2 目标用户
- 加密货币交易者
- 投资分析师
- 代币研究人员
- 量化交易用户

## 2. 用户故事和验收标准（EARS语法）

### 2.1 核心功能用户故事

#### 故事1：涨幅榜切换
**用户故事**: 作为一个交易者，我希望能够在热搜榜和涨幅榜之间切换，以便查看不同维度的代币数据。

**验收标准**:
- **Given** 用户在主页面上
- **When** 用户点击涨幅榜标签
- **Then** 页面应该显示涨幅榜数据
- **And** 涨幅榜标签应该被高亮显示
- **And** 数据应该在10秒内自动刷新

#### 故事2：多链数据展示
**用户故事**: 作为一个多链交易者，我希望能够查看不同区块链的涨幅数据，以便做出投资决策。

**验收标准**:
- **Given** 用户在涨幅榜页面
- **When** 用户选择特定链（如Solana、BSC等）
- **Then** 页面应该显示该链的涨幅数据
- **And** 数据应该包含所有必要的字段信息
- **And** 筛选应该在2秒内完成

#### 故事3：时间间隔筛选
**用户故事**: 作为一个短线交易者，我希望能够选择不同的时间间隔来查看涨幅数据，以便进行短期交易决策。

**验收标准**:
- **Given** 用户在涨幅榜页面
- **When** 用户选择时间间隔（1m、5m、15m、1h、4h、24h）
- **Then** 涨幅数据应该按选择的时间间隔显示
- **And** 价格变化列应该动态更新
- **And** 排序应该基于选择的时间间隔

#### 故事4：数据字段显示
**用户故事**: 作为一个专业交易者，我希望能够看到完整的代币信息，包括流动性、市值、持有者等数据，以便进行全面分析。

**验收标准**:
- **Given** 用户在涨幅榜页面
- **When** 用户查看代币列表
- **Then** 每个代币应该显示以下默认字段：
  - Pair/Age（代币对/时间）
  - AI叙事
  - Mc（市值）
  - Liq/initial（流动性/初始）
  - Price（价格）
  - 1m%（1分钟涨幅）
  - 15m%（15分钟涨幅）
  - 24h%（24小时涨幅）
  - 24h Volume/Txn（24小时交易量/交易数）
  - Markers（标记）
  - Holders（持有者）
  - Smarters（智能钱包）
  - Security（安全性）
  - DEX
  - Operation（操作）

#### 故事5：实时数据更新
**用户故事**: 作为一个活跃交易者，我希望涨幅数据能够实时更新，以便及时抓住交易机会。

**验收标准**:
- **Given** 用户在涨幅榜页面
- **When** 有新的价格数据更新
- **Then** 相关代币的涨幅数据应该立即更新
- **And** 排序应该动态调整
- **And** 更新不应该影响用户的当前操作

#### 故事6：收藏功能
**用户故事**: 作为一个重度用户，我希望能够收藏感兴趣的代币，以便快速跟踪。

**验收标准**:
- **Given** 用户已登录且在涨幅榜页面
- **When** 用户点击代币的收藏按钮
- **Then** 该代币应该被添加到收藏列表
- **And** 收藏状态应该被持久化保存
- **And** 应该显示收藏成功的提示

#### 故事7：页面导航
**用户故事**: 作为一个用户，我希望能够点击代币进入详情页，以便查看更多信息。

**验收标准**:
- **Given** 用户在涨幅榜页面
- **When** 用户点击任意代币行
- **Then** 应该跳转到该代币的详情页面
- **And** 详情页应该包含完整的代币信息
- **And** 导航应该在新页面中打开

### 2.2 高级功能用户故事

#### 故事8：黑名单过滤
**用户故事**: 作为一个风险管理者，我希望能够过滤掉黑名单中的代币，以便避免风险投资。

**验收标准**:
- **Given** 用户启用了黑名单过滤
- **When** 涨幅榜加载数据
- **Then** 黑名单中的代币应该被自动过滤
- **And** 过滤应该基于合约地址和关键词
- **And** 过滤状态应该被保存

#### 故事9：快速交易
**用户故事**: 作为一个快速交易者，我希望能够直接在涨幅榜上进行快速交易，以便抓住交易机会。

**验收标准**:
- **Given** 用户启用了快速交易功能
- **When** 用户在涨幅榜上进行交易操作
- **Then** 应该弹出快速交易界面
- **And** 交易应该能够快速执行
- **And** 交易状态应该实时反馈

## 3. 业务逻辑

### 3.1 数据获取逻辑
1. **API调用**: 使用 `getTreasureList` API，传入 `category: 'gain'` 参数
2. **数据映射**: 通过 `listMapFunction` 对原始数据进行处理和映射
3. **实时更新**: 通过WebSocket连接获取价格更新数据
4. **排序逻辑**: 根据选择的时间间隔和涨幅进行排序

### 3.2 筛选逻辑
1. **链筛选**: 根据选择的区块链过滤数据
2. **时间间隔筛选**: 根据选择的时间间隔显示对应的涨幅数据
3. **黑名单过滤**: 过滤掉用户设置的黑名单代币
4. **分页处理**: 支持分页显示，默认每页50条数据

### 3.3 UI交互逻辑
1. **标签切换**: 在热搜榜和涨幅榜之间切换
2. **实时更新**: 每10秒自动刷新数据
3. **排序交互**: 支持按不同字段排序
4. **响应式设计**: 适配不同屏幕尺寸

## 4. 技术要求

### 4.1 性能要求
- 数据加载时间 < 2秒
- 实时更新延迟 < 1秒
- 支持同时显示500+代币数据
- 内存使用优化，避免内存泄漏

### 4.2 兼容性要求
- 支持主流现代浏览器
- 响应式设计，支持移动端
- 支持暗黑/明亮主题切换
- 支持多语言国际化

### 4.3 安全要求
- 数据传输加密
- 防止XSS攻击
- 用户输入验证
- 钱包连接安全验证

## 6. 风险评估

### 6.1 技术风险
- **数据量过大**: 可能导致页面性能问题
- **WebSocket连接**: 可能出现连接不稳定
- **API限流**: 可能影响数据获取频率

### 6.2 业务风险
- **数据准确性**: 涨幅数据的准确性关系到用户决策
- **实时性要求**: 数据延迟可能影响交易时机
- **用户体验**: 复杂的数据展示可能影响用户使用

### 6.3 风险缓解措施
- 实施数据缓存策略
- 添加WebSocket重连机制
- 实施API请求限流和重试机制
- 添加数据验证和错误处理
- 实施性能监控和优化
